Aeons: The Grand Unfolding — Developer Tutorial

Welcome, new team member! This guide explains how the game works, how the code is organised, and how to make common changes. Read it from top to bottom on your first day, then use it as a reference afterwards.

To export as PDF: open tutorial.html in any modern browser and press Ctrl+P (or ⌘+P on Mac) → set destination to Save as PDF → Save.

Welcome

Aeons: The Grand Unfolding is a browser-based incremental (idle) game built entirely in vanilla JavaScript with zero external runtime dependencies. There is no build step, no bundler, and no framework. What you see in the repo is exactly what runs in the browser.

The game follows the player character through the birth and evolution of the universe — from absorbing raw quantum energy motes in the void, through stellar fusion, elemental chemistry, and eventually quark physics and subatomic matter. Progress is gated by upgrades, milestones, and periodic prestige resets that carry permanent bonuses forward.

Running the Game Locally

ES Modules require a server — browsers block import statements from file:// URLs due to CORS restrictions.

Option 1 — Python (recommended, zero setup)

cd /path/to/IncrementalGame
python -m http.server 8080
# Open: http://localhost:8080

Option 2 — VS Code Live Server

  1. Install the Live Server extension.
  2. Right-click index.html → Open with Live Server.
  3. The browser opens at http://127.0.0.1:5500 and auto-reloads on file save.

Option 3 — Node.js serve

npx serve . -p 8080

There is no build step — no npm run build, no compiled output. Edit a .js file and reload the browser. Cache-busting query strings (?v=SHA) are added automatically by the deploy workflows.

Game Overview

Understanding the game loop helps you understand the code.

Phase 1 — Energy (the void)

The player starts as a particle in empty space. A single glowing beacon mote drifts toward the player from the right. When it makes contact, the player earns 1 energy — enough to buy the first upgrade, Cosmic Drift, which unlocks movement.

Once Electromagnetic Bond is purchased (cost: 10 energy), the void fills with ambient motes that are passively pulled toward the player. Each absorbed mote awards energy. Energy is capped at 500 initially; upgrades raise the cap up to 10,000,000.

Visual progression: energy is displayed as orbiting motes around the player character — one mote per unit, organised into 7 tiers by order of magnitude (see Orbital Energy Display).

Phase 2 — Prestige / Aeons

When the current energy cap is fully reached, a "Void Saturated" prompt appears. Pressing it enters the Prestige panel. The player earns Aeons (a meta-currency) proportional to peak energy, spends them on permanent cross-run upgrades, then starts a new run with a higher energy cap.

After the first prestige, Dark Matter nodes spawn in the void; the player must navigate to them to collect Dark Matter, which feeds into synthesis chains.

Phase 3 — Epoch Collapse & Quarks

At 10,000,000 energy the game automatically triggers an Epoch Collapse cinematic — all particles are drawn into the player, the screen flashes, and the quark phase begins. The player now allocates percentages across six quark flavours (up, down, strange, charm, bottom, top), each giving a different passive multiplier. Orbital motes shift colour to blend the quark palette.

Phase 4 — Subatomic

After a second Epoch Collapse, the Particle Forge opens. Quarks are consumed to synthesise protons, neutrons, and electrons. These become the new orbital display — replacing energy motes with particle rings.

The golden rule: data flows downward through the layers. UI listens to EventBus events emitted by engines. Engines never touch the DOM. The data layer contains only plain objects.

Core Systems

EventBus

File: js/core/EventBus.js

The EventBus is the primary communication channel between systems. It is a synchronous pub/sub singleton — listeners fire immediately in registration order when an event is emitted.

// Subscribe
EventBus.on('resource:updated', ({ resourceId, newValue, delta }) => {
  console.log(`${resourceId} is now ${newValue}`);
});

// Publish
EventBus.emit('resource:updated', {
  resourceId: 'energy',
  newValue: 42,
  delta: 1,
  ratePerSec: 5,
});

// Unsubscribe
EventBus.off('resource:updated', myCallback);

Key events used across the codebase:

EventEmitterPayload
resource:updatedResourceManager{ resourceId, newValue, delta, ratePerSec }
upgrade:purchasedUpgradeSystem{ upgradeId, level, definition }
upgrade:affordableUpgradeSystem{ upgradeId, canAfford }
milestone:triggeredMilestoneSystem{ milestoneId, definition }
particle:absorbedCanvasRenderer{ quality, x, y }
prestige:executePrestigeSystem{ count, aeonsEarned, aeonTotal, peakEnergy }
prestige:purgatory:enterPrestigePanel—
prestige:purgatory:exitPrestigePanel—
collapse:startmain.js—
collapse:completemain.js—
quarks:colorChangedQuarkEngine{ color }
dm:nodeCollectedDarkMatterSystem{ x, y, value }

Never bypass the EventBus. If system A needs to trigger behaviour in system B, it emits an event. System B subscribes. This prevents circular imports and tight coupling.

GameLoop

File: js/core/GameLoop.js

The GameLoop runs via requestAnimationFrame and provides two callback registrations:

  • onTick(cb) — fires at a fixed rate (default 20 Hz / 50 ms). Used for game logic: resource generation, milestone checks, particle physics. Receives dt in seconds.
  • onFrame(cb) — fires at full rAF rate (typically 60 fps). Used for canvas rendering. Receives a DOMHighResTimeStamp.

The tick accumulator pattern decouples simulation rate from render rate:

// Internal loop (simplified):
accumulator += wallDelta;
while (accumulator >= tickInterval) {
  accumulator -= tickInterval;
  const adjustedDt = tickInterval * (window.AEONS_SPEED_MULT ?? 1);
  for (const cb of tickCallbacks) cb(adjustedDt);
}

AEONS_SPEED_MULT is the dev speed multiplier (1×/10×/100×/1000×). Setting it affects only the adjustedDt passed to tick callbacks, so all game logic sees the multiplied time but real-world elapsed time is unchanged.

// To register callbacks (done once in main.js):
GameLoop.onTick((dt) => resourceManager.tick(dt));
GameLoop.onFrame((timestamp) => canvasRenderer.onFrame(timestamp));
GameLoop.start();

ResourceManager

File: js/engine/ResourceManager.js

Owns all resource state. A resource has a definition (from epoch1.js, static) and a state (live, mutated each tick).

Resource state object:

{
  id: 'energy',
  currentValue: 0,          // current amount
  passiveRatePerSec: 5,     // passive generation per second
  cap: 500,                 // soft cap (currentValue won't exceed this)
  displayLabel: 'Energy',
  visible: true,
  derived: false,           // true = calculated from other resources, not generated
}

Key methods:

resourceManager.get('energy')          // → resource state object
resourceManager.add('energy', 10)      // add amount (respects cap)
resourceManager.set('energy', 0)       // force-set (used for reset)
resourceManager.getAll()               // → array of all resource states
resourceManager.tick(dt)               // advances passive generation
resourceManager.recalculateRates()     // re-reads all upgrade effects
resourceManager.applyCapBonus('energy', 500)   // add persistent cap bonus

Rate calculation happens in recalculateRates(). It queries upgradeSystem.getPurchasedEffects() and applies each effect according to its effectType:

  • rateMultiplier — multiplies the passive rate
  • absorptionMultiplier — applied per-mote when a particle is absorbed
  • capMultiplier — multiplies the resource cap

UpgradeSystem

File: js/engine/UpgradeSystem.js

Manages the upgrade shop: definitions, purchase logic, prerequisite validation, cost scaling.

Upgrade definition (from epoch1.js):

{
  id: 'upg_gravitationalPull',
  name: 'Electromagnetic Bond',
  baseCost: 10,              // cost of level 1
  costResource: 'energy',   // which resource is spent
  maxLevel: 10,             // how many times purchasable
  costScaling: 3.0,         // multiplier per level (exponential)
  hardScalingStart: 5,      // level at which steeper scaling kicks in
  hardScaling: 5.0,         // multiplier for levels > hardScalingStart
  prerequisites: ['upg_cosmicDrift'],   // must be purchased first
  requiresMilestone: null,  // optional milestone gate
  effectType: 'unlock',
  effectTarget: 'attraction',
  effectMagnitude: 1,
}

Cost formula:

cost(level) = baseCost × costScaling^(level - 1)   // levels 1..hardScalingStart
cost(level) = cost(hardScalingStart) × hardScaling^(level - hardScalingStart)  // above threshold

Key methods:

upgradeSystem.canPurchase('upg_gravitationalPull')   // → boolean
upgradeSystem.getLockReason('upg_gravitationalPull') // → string or null
upgradeSystem.purchase('upg_gravitationalPull')      // → boolean (success)
upgradeSystem.getCost('upg_gravitationalPull')        // → number
upgradeSystem.getLevel('upg_gravitationalPull')       // → 0..maxLevel
upgradeSystem.forceLevel('upg_cosmicDrift', 1)        // bypass cost check (used post-prestige)

When purchased, the upgrade emits upgrade:purchased on the EventBus. All systems that care about an upgrade's effect subscribe to this event.

MilestoneSystem

File: js/engine/MilestoneSystem.js

Milestones are one-shot events that fire when a condition is met. They drive the narrative, unlock UI elements, and apply rewards.

Milestone definition (from epoch1.js):

{
  id: 'ms_gasCloud',
  title: 'The Gas Cloud Stirs',
  flavourText: 'Dust and debris coalesce...',
  conditionType: 'resource_threshold',
  conditionTarget: 'mass',
  conditionValue: 100,
  reward: { type: 'unlockResource', target: 'darkMatter' },
}

MilestoneSystem.check() is called every tick. It evaluates each un-triggered milestone and fires if its condition is met. When triggered, it emits milestone:triggered on the EventBus, which the MilestoneNotification UI catches to show a toast.

SaveSystem

File: js/core/SaveSystem.js

Persists all game state to localStorage under the key aeons_save_v1. Auto-saves every 60 seconds.

Save payload structure:

{
  schemaVersion: 3,          // used by SaveMigrator for backward compatibility
  savedAt: Date.now(),
  gameState: { ... },        // aeonCount, prestigeCount, collapsedOnce, …
  resourceStates: { ... },   // currentValue, passiveRatePerSec, cap per resource
  upgradeStates: { ... },    // purchased, level per upgrade
  milestoneStates: { ... },  // triggered per milestone
  quarkState: { ... },       // allocation percentages
  prestigeState: { ... },    // banked aeons, purchased prestige upgrades
  ...
}

Save encoding: Uses TextEncoder/TextDecoder → btoa for UTF-8-safe Base64. This handles Unicode chars in resource labels (e.g. H₂O).

Migrations (SaveMigrator.js): when a save is loaded with an older schemaVersion, migration functions update it to the current schema before use. Always increment CURRENT_VERSION and add a migration function when adding new save fields.

// Pattern for a new save field:
// In SaveMigrator.js:
4: (data) => {
  if (!data.myNewField) data.myNewField = defaultValue;
  return data;
}

Rendering

CanvasRenderer

File: js/renderer/CanvasRenderer.js

The master renderer orchestrates all visual layers. It registers an onFrame callback with the GameLoop and draws everything each animation frame.

Draw order each frame:

  1. Clear both canvases
  2. Draw space dust background (parallax)
  3. Draw region backgrounds (void gradient)
  4. Draw dark matter node halos
  5. Draw DM gravity waves
  6. Draw ambient particles (ParticleSystem)
  7. Draw dark matter nodes
  8. Draw the player "home object" (player character + attraction rings + orbital motes)
  9. Draw DM reflected ripples
  10. Draw floating numbers (+energy values)
  11. Draw minimap (inset, top-right)
  12. Draw glow pass (bloom canvas)
  13. Draw dev speed indicator (if active)

The player character is drawn at the camera centre (world coordinates moteController.worldX/Y). The orbital energy motes orbit around this same point.

Camera

File: js/renderer/Camera.js

A simple 2D pan-only camera. The game world is flat (no zoom).

// World to screen:
const { sx, sy } = camera.worldToScreen(worldX, worldY);
// sx = worldX - camera.x   (no scaling)
// sy = worldY - camera.y

The camera follows the player: camera.x = moteController.worldX - canvas.width/2.

ParticleSystem

File: js/renderer/ParticleSystem.js

Manages up to ~250 ambient void motes. Each particle has position, velocity, quality level (0–4), and a lifetime. Particles outside the attraction zone wander randomly; inside, they decelerate toward the player following a quartic speed falloff (t^4 where t = 1 - dist/radius), creating a natural fast-inner / slow-outer effect.

When a particle reaches the player (within contact radius ~10 px), it emits particle:absorbed and is removed. main.js handles this event and adds energy.

OrbitalEnergyDisplay

File: js/renderer/OrbitalEnergyDisplay.js

Draws the 7-tier orbital mote ring around the player. Each tier corresponds to a power of 10 in current energy. Motes per tier = the leading decimal digit at that order of magnitude (1–9).

Mote insertion and spacing:

  • New motes are placed at the midpoint of the largest angular gap in the existing ring, maintaining smooth even spacing without animation restarts.
  • When motes are removed, remaining motes preserve their angles (truncate from end).
  • As motes spread evenly around the tier, each individual mote may slightly slow its rotation until proper spacing is achieved, then returns to full tier speed.

Mote recycling: To prevent hitting the 500-mote-per-region density cap during infinite canvas exploration, the system recycles the furthest 10% of motes each generation tick (when spawning is active and count > 50). This allows distant motes to be freed and respawned closer to the player.

Game Engines

DarkMatterSystem

File: js/engine/DarkMatterSystem.js

Unlocked at milestone ms_gasCloud. Spawns invisible DM nodes at random void positions. Each node pulses gravity waves (expanding circles at 360 px/s, sized to just reach the player). When a wave front hits the player, a reflected ripple travels back toward the node in a 60° cone.

Player collects a node by physically moving within the collection radius (default 60 px, increased by prs_dmSiphon prestige upgrade).

PrestigeSystem

File: js/engine/PrestigeSystem.js

Manages two prestige currencies:

  • Aeons — earned on each prestige, scales with peak energy. Used for meta-upgrades (rate bonuses, cap expansion, automation).
  • Epoch Echoes — awarded by Epoch Collapses. Used for quark and subatomic upgrades.

Prestige flow:

Player clicks "Ascend to the Void"
  → PrestigePanel.executePrestige() emits prestige:execute
    → main.js stores pendingPrestigeData
  → PrestigePanel emits prestige:purgatory:enter
    → main.js resets all run systems, reloads epoch
    → PrestigeSystem.applyRunBonuses() grants free upgrades, seed energy, movement unlock
  → Player closes prestige menu
    → PrestigePanel emits prestige:purgatory:exit
      → main.js unblocks input, starts new run

The purgatory state (gameState.prestigePurgatory = true) keeps the game reset but doesn't start the new run until the player closes the prestige menu. This allows the player to spend Aeons before committing to the next run.

FusionEngine

File: js/engine/FusionEngine.js

Runs hydrogen-to-helium fusion in active stars:

  • Main sequence (H → He): runs when a star is in main_sequence stage
  • Red giant (He → C/O): runs when a star reaches red_giant stage
  • Supernova (Fe burst): runs when a star collapses

FusionEngine is ticked by the GameLoop and checks upgrade levels to scale reaction rates.

QuarkEngine

File: js/engine/QuarkEngine.js

After the first Epoch Collapse, the player allocates 100% of "quark charge" across six flavours. Each flavour gives a passive multiplier. QuarkEngine computes a blended colour from allocations and emits quarks:colorChanged so the orbital display updates mote colours.

QuarkColourEffect
Up#ff6666+% energy absorption rate
Down#6666ff+% mote spawn rate
Strange#66ff66+% dark matter collection
Charm#ffcc44+% fusion rate
Bottom#cc44ff+% molecule synthesis
Top#ffd700+% EM Bond attraction radius

UI Panels

Each UI panel is a class that:

  1. Takes a reference to EventBus (and sometimes engine instances) in its constructor
  2. Has an init() method that queries DOM elements and subscribes to relevant EventBus events
  3. Renders itself by updating DOM when it receives events

Anatomy of a typical panel:

export class ResourcePanel {
  #eventBus;
  #resourceManager;
  #listEl;

  constructor(EventBus) {
    this.#eventBus = EventBus;
  }

  setResourceManager(rm) {
    this.#resourceManager = rm;
  }

  init() {
    this.#listEl = document.getElementById('resource-list');

    // Subscribe to resource updates — re-render on every change
    this.#eventBus.on('resource:updated', ({ resourceId }) => {
      this.#renderResource(resourceId);
    });
  }

  #renderResource(id) {
    const state = this.#resourceManager.get(id);
    // ... update DOM ...
  }
}

Key rule: UI panels must never call engine methods directly to modify game state. They should only read state (via engine getters) and listen to events.

Writing a Unit Test

// tests/unit/my-feature.test.js
import { test, expect } from '@playwright/test';
import { setupGame } from '../helpers/game-helpers.js';

test.describe('Electromagnetic Bond', () => {
  let g;

  test.beforeEach(async ({ page }) => {
    g = await setupGame(page);
  });

  test('unlocks attraction after purchase', async () => {
    // Give enough energy to buy both prerequisites
    await g.setResource('energy', 100);

    // Buy Cosmic Drift first (prerequisite)
    await g.purchaseUpgrade('upg_cosmicDrift');

    // Buy EM Bond
    const result = await g.purchaseUpgrade('upg_gravitationalPull');
    expect(result.success).toBe(true);

    // Verify the upgrade state
    const upg = await g.getUpgrade('upg_gravitationalPull');
    expect(upg.level).toBe(1);
  });

  test('costs scale exponentially', async () => {
    await g.setResource('energy', 999999);
    await g.purchaseUpgrade('upg_cosmicDrift');

    const costs = [];
    for (let i = 0; i < 5; i++) {
      const upg = await g.getUpgrade('upg_gravitationalPull');
      costs.push(upg.cost);
      await g.purchaseUpgrade('upg_gravitationalPull');
    }

    // Each cost should be ~3× the previous (costScaling: 3.0)
    for (let i = 1; i < costs.length; i++) {
      expect(costs[i] / costs[i - 1]).toBeCloseTo(3.0, 0);
    }
  });
});

How To Guides

Add a New Resource

Step 1 — Define it in js/data/epoch1.js:

// In the resources array:
{
  id: 'plasma',
  displayLabel: 'Plasma',
  epochId: 'epoch1',
  initialValue: 0,
  initialRate: 0,
  cap: null,        // null = uncapped
  unit: '',
  description: 'Superheated ionised matter.',
  visible: false,   // hidden until a milestone reveals it
  derived: false,
}

Step 2 — Reveal it via a milestone reward:

// In the milestones array:
{
  id: 'ms_plasmaDiscovered',
  conditionType: 'resource_threshold',
  conditionTarget: 'energy',
  conditionValue: 5000,
  reward: { type: 'unlockResource', target: 'plasma' },
  ...
}

Step 3 — Generate it somewhere. Either set initialRate > 0, or add an upgrade that changes the rate, or generate it in an engine's tick() method:

// Example: generate 0.1 plasma/sec when an upgrade is active
resourceManager.add('plasma', 0.1 * dt);

That's it. ResourceManager initialises all resources from the epoch config automatically. ResourcePanel renders all visible resources.

Add a New Upgrade

Step 1 — Add to js/data/epoch1.js:

// In tier1Upgrades (or whichever tier array):
{
  id: 'upg_plasmaIgnition',
  epochId: 'epoch1',
  name: 'Plasma Ignition',
  description: 'Ionise absorbed motes into plasma.',
  category: 'tier1',
  baseCost: 1000,
  costResource: 'energy',
  maxLevel: 5,
  costScaling: 4.0,
  tier: 2,
  prerequisites: ['upg_vacuumHarvesting'],
  effectType: 'spawnRate',      // or a custom type you define
  effectTarget: 'plasma',
  effectMagnitude: 2.0,
  requiresMilestone: 'ms_plasmaDiscovered',
  mechanic: '⚙ ×2.0 plasma per level',
}

Step 2 — Handle the effect. If your effectType is already handled (e.g. absorptionMultiplier in ResourceManager.recalculateRates()), nothing else is needed. For a new effect type:

// In main.js, inside the upgrade:purchased event listener:
if (data.upgradeId === 'upg_plasmaIgnition') {
  const level = upgradeSystem.getLevel('upg_plasmaIgnition') || 0;
  // apply effect...
}

Or add a handler in ResourceManager.recalculateRates() if it is a continuous rate effect:

// In ResourceManager.recalculateRates():
case 'plasmaMultiplier':
  plasmaRate *= Math.pow(def.effectMagnitude, state.level);
  break;

Step 3 — Write a test (see Write a Unit Test).

Pre-Commit Checklist

Before committing any changes, always:

  1. Lint syntax — catches parse errors before reaching browser:

``bash npm run lint ` This runs scripts/syntax-check.js which parses all .js files. Fix any SyntaxError` output before proceeding.

  1. Test locally — play the game for 30 seconds after your changes:

``bash python -m http.server 8080 # Open http://localhost:8080 and verify basic gameplay `` Common issues:

  • White screen → likely a parse/import error (check DevTools Console)
  • Broken UI → likely a rendering or DOM issue (check DevTools Elements)
  • Logic broken → likely a state or event issue (check DevTools Console for error logs prefixed with [Module])
  1. No regressions — if you changed:
  • Upgrade system → verify upgrades still purchase and apply effects
  • Rendering → verify the player, particles, and UI render without visual artifacts
  • Save/load → verify page reload preserves game state
  • Events → verify triggered events still fire and are handled (search logs for [Module] event:name)
  1. Optional: run unit tests — if you have time:

``bash npm test `` Note: Tests are slow (~5 min). Only run if you modified core logic (not UI tweaks or visual changes).

If you skip these checks and break the game:

  • Developers have to revert your commit (wasted time)
  • Beta users encounter broken builds (poor experience)
  • You lose credibility with the team (matters for code reviews)

Minimum viable pre-commit:

npm run lint
# Reload browser — verify no white screen or obvious errors
